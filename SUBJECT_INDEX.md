# ARRP Subject and Institution Index

This is a living discovery index for finding the ARRP area or issue that owns a subject, government body, or other institution. It supplements the ordered [Project Areas index](areas/README.md); it does not create new issue boundaries or replace an issue page, area page, the [ARRP GitHub Project](https://github.com/users/Thorncrag/projects/2), or the [GitHub issue registry](inventory/github_issue_registry.csv).

The index is intentionally many-to-many. An organization may appear in several proposals, and a subject may cross several areas. **Primary** identifies the project's principal analytical home. **Related** identifies a material implementation, manifestation, or boundary connection. The index does not report proposal status or score because those fields change independently and belong in the GitHub Project and issue audit records.

In a compiled print edition, the stable identifiers and Markdown links below are source keys for edition-specific page locators. The export process should resolve each linked area or issue to its page or page range and render those locators in a back-of-book index. Page numbers are not maintained in this canonical file because they change with content, edition, trim size, and layout.

This first edition emphasizes developed proposals and high-value candidate routing. It is not yet exhaustive. New proposal development and material source integration should add useful plain-language terms, acronyms, aliases, and institutional names while the relevant context is fresh.

## Maintenance Standard

Update this index when work creates or materially changes any of the following:

1. a developed issue or a stable candidate issue that supplies a useful subject route;
2. the primary home or a material cross-reference for an existing subject;
3. a department, agency, office, court, international body, State institution, acronym, or common alias that a reader could reasonably use to find relevant work; or
4. a manifestation that makes an existing organization materially relevant to a proposal for the first time.

Keep entries short and navigational. Link to the most specific stable project page available, identify the primary home when one exists, and use related links instead of reproducing issue analysis. Do not copy lifecycle status, score, priority, audit tier, or other volatile GitHub Project fields into this index.
